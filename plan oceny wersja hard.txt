🔐 OCENA 3 — 2FA (TOTP)

Użytkownik włącza 2FA i loguje się kodem z Google Authenticator.

Dodaj biblioteki

npm i speakeasy qrcode


Zmiany w modelu User

twoFactorEnabled – czy 2FA jest aktywne

twoFactorSecret – stały sekret TOTP

twoFactorTempSecret – używany przy aktywacji

Nowe endpointy:

POST /2fa/start → generuje sekret i QR code (dataURL)

POST /2fa/verify → użytkownik podaje kod, backend go weryfikuje

Logowanie z 2FA

Jeśli user ma 2FA → musi podać kod TOTP razem z loginem.

Brak kodu = 401 “2FA required”.

Frontend

Panel "Security" po zalogowaniu: przycisk "Enable 2FA"

Pokazujesz QR i input

W loginie – jeśli backend żąda kodu 2FA, pokazujesz input

🧠 OCENA 4 — Lepsze szyfrowanie (AES-GCM + klucz użytkownika)

Każdy użytkownik ma własny klucz, zaszyfrowany masterem.

W .env:

MASTER_KEY_BASE64=...  // 32 bajty zakodowane base64


Przy rejestracji użytkownika:

Tworzysz losowy userKey (32 bajty)

Szyfrujesz userKey masterowym kluczem

Zapisujesz do usera:

vaultKeyEnc, vaultKeyIv, vaultKeyTag

Szyfrowanie wpisów vault

JSON z danymi (title, login, url, itd.) → zamieniasz na payload

Szyfrujesz payload userKeyem → zapisujesz do bazy:

dataEnc, iv, tag

Odczyt:

Deszyfrujesz userKey z danych usera

Potem deszyfrujesz wpisy

Uwaga: Nie musisz szyfrować każdego pola osobno – cały JSON też jest OK.

🔒 OCENA 5 — Rate limiting + lockout

Zabezpieczenie przed spamem i atakami brute-force.

Rate limiting (na IP)

npm i express-rate-limit

import rateLimit from "express-rate-limit";

const limiter = rateLimit({
  windowMs: 60 * 1000,
  max: 10,
  message: { message: "Too many login attempts" },
});

app.post("/auth/login", limiter, loginHandler);


Lockout (na użytkownika)

W modelu User:

failedLoginAttempts

lockUntil

Przy loginie:

Złe hasło → +1

5 prób = lockUntil = teraz + 15min

Udany login → reset liczników

✅ OCENA 6 — Testy + CI (GitHub Actions)

Tu wchodzisz do prawdziwego świata DevOps i QA.

Zainstaluj testy

npm i -D jest supertest mongodb-memory-server


Struktura testów

Folder backend/tests/

Pliki testowe: auth.test.js, vault.test.js, itd.

Importuj app z app.js (nie server.js – nie otwieraj portu)

W testach ustaw process.env

JWT_SECRET, MASTER_KEY_BASE64 – możesz dać stałe wartości

Skrypt testowy w package.json

"scripts": {
  "test": "jest --runInBand"
}


CI: GitHub Actions

Plik .github/workflows/ci.yml

name: CI

on:
  push:
  pull_request:

jobs:
  test-backend:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install backend deps
        working-directory: backend
        run: npm ci

      - name: Run tests
        working-directory: backend
        env:
          JWT_SECRET: test
          MASTER_KEY_BASE64: AAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAAA=
        run: npm test


Efekt:

W repo na GitHubie → zakładka Actions → zielony ✅

W README.md możesz dodać badge z status CI
